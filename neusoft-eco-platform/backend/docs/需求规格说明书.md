## 一、引言

### 1.1 编写目的

本需求规格说明书旨在全面、准确地描述"东软环保公众监督平台"的功能需求、非功能需求及业务流程，为系统设计、开发、测试和验收提供依据。本文档面向项目开发人员、测试人员、系统管理员及项目相关决策者。

### 1.2 项目范围

本系统覆盖空气质量公众监督全流程，包括公众监督员反馈提交、网格员实地核查、管理员任务指派与决策者数据统计四大核心业务环节。系统面向省级环保监管部门，实现从公众反馈到政府响应的闭环监管。

### 1.3 术语与缩写

| 术语/缩写 | 说明 |
|-----------|------|
| AQI | Air Quality Index，空气质量指数 |
| PM2.5 | 细颗粒物，空气动力学当量直径≤2.5μm的颗粒物 |
| PM10 | 可吸入颗粒物，空气动力学当量直径≤10μm的颗粒物 |
| 网格员 | 负责特定网格区域的空气质量监测工作人员 |
| 公众监督员 | 注册后可提交空气质量反馈的普通公众用户 |
| 闭环监管 | 从发现问题→指派任务→实地核查→数据确认的完整业务闭环 |
| SSM | Spring + SpringMVC + MyBatis 框架组合 |
| Spring Boot | Spring 生态的快速开发框架，本项目采用2.7.14版本 |
| MyBatis-Plus | MyBatis增强工具，简化CRUD操作 |

## 二、项目概述

### 2.1 项目背景

近年来，随着工业化和城市化进程加速，空气质量问题成为公众关注的焦点。传统的环境监测模式依赖固定站点采集数据，存在监测覆盖面有限、响应滞后、公众参与度低等问题。

国家《生态环境监测网络建设方案》明确提出"政府主导、部门协同、社会参与、公众监督"的环境监测体系建设要求。在此背景下，亟需建设一个打通公众监督与政府监管的数字化平台，让公众成为环境治理的参与者和监督者。

东软环保公众监督平台正是在此需求下规划建设的，旨在通过信息化手段构建"公众反馈—政府响应—网格核查—数据决策"的闭环监管体系，提升环保监管的时效性、透明度和公众参与度。

### 2.2 建设目标

#### 2.2.1 总体目标

建设覆盖省—市—网格三级体系的空气质量公众监督平台，实现公众监督员随手反馈、系统智能指派、网格员实地核查、决策者数据驱动的闭环监管模式。

#### 2.2.2 具体目标

1. **公众参与便捷化**：公众监督员可通过系统快速提交所在区域的空气质量反馈，支持AQI及六项污染物指标录入
2. **任务指派自动化**：系统管理员可高效将公众反馈指派给对应网格区域的网格员，减少人工调度成本
3. **核查流程标准化**：网格员实地监测后在线确认数据，形成公众反馈与实测数据的对比链路
4. **决策支持可视化**：为决策者提供多维度的AQI统计与趋势分析，支撑环保政策制定
5. **数据管理规范化**：建立统一的空气质量数据库，实现数据标准化采集、存储与分析

### 2.3 系统边界

本系统为后端服务平台，提供RESTful API接口。前端展示层（Web/移动端）不在本期开发范围内，但接口设计需兼顾多端对接需求。

## 三、角色定义与权限

### 3.1 角色总览

系统定义四类核心角色，各角色在闭环监管流程中承担不同职责：

| 角色 | 标识 | 核心职责 | 操作范围 |
|------|------|---------|---------|
| 公众监督员 | public_supervisor | 提交空气质量反馈 | 反馈提交、个人记录查看 |
| 网格员 | grid_member | 实地核查与数据确认 | 任务查询、实测确认 |
| 系统管理员 | sys_admin | 反馈指派与系统管理 | 反馈指派、人员管理 |
| 决策者 | decision_maker | 统计分析与决策 | 数据统计、趋势分析 |

### 3.2 角色详细描述

#### 3.2.1 公众监督员

**身份说明**：经注册认证的普通公众用户，是环保监督的"前哨"。

**业务权责**：
- 在发现空气质量异常时，提交所在网格区域的AQI反馈信息
- 反馈内容包含网格地址、AQI数值及可选的六项污染物浓度数据
- 可查看个人历史反馈记录及处理状态
- 不可修改已提交的反馈内容

**操作约束**：
- 每次反馈必须指定网格地址
- AQI数值为必填项，污染物指标为选填项
- 反馈提交后状态为"待指派"，不可自行撤回

#### 3.2.2 网格员

**身份说明**：负责特定网格区域空气质量监测的工作人员，是核查的"执行者"。

**业务权责**：
- 查看被指派的反馈任务列表
- 实地监测后提交实测AQI数据
- 对比公众反馈与实测数据，确认是否一致
- 每条反馈仅可确认一次

**操作约束**：
- 仅可查看指派给自己的任务
- 确认结果为"一致"或"不一致"二选一
- 实测数据提交后不可修改

#### 3.2.3 系统管理员

**身份说明**：平台运营管理人员，是任务的"调度中枢"。

**业务权责**：
- 查看待指派的公众反馈
- 将反馈指派给对应网格区域的网格员
- 管理网格员账号的启用/禁用状态
- 维护省市区基础数据

**操作约束**：
- 每条反馈仅可指派给一名网格员
- 已指派的反馈不可更改指派对象
- 区分超级管理员与普通管理员角色

#### 3.2.4 决策者

**身份说明**：环保部门决策层人员，是数据的"消费者"。

**业务权责**：
- 按时间维度查看AQI统计数据（日均/月均/季度均）
- 按区域维度查看AQI统计数据（省级/市级）
- 查看AQI趋势变化曲线
- 获取汇总统计指标（平均值、最大值、最小值）

**操作约束**：
- 仅可查看统计数据，不可修改原始数据
- 统计维度可选，默认为日维度

### 3.3 角色与业务流程关系

```
公众监督员 ──[提交反馈]──> 系统管理员 ──[指派任务]──> 网格员 ──[确认实测]──> 决策者
     │                        │                        │                    │
     │                        │                        │                    │
   发现问题                 调度分配                 实地核查             数据决策
```

## 四、功能性需求

### 4.1 功能需求总览

| 编号 | 功能模块 | 子功能 | 优先级 | 对应角色 |
|------|---------|--------|--------|---------|
| FR-01 | 公众反馈 | 提交AQI反馈 | 高 | 公众监督员 |
| FR-02 | 公众反馈 | 查看反馈状态 | 中 | 公众监督员 |
| FR-03 | 任务指派 | 查看待指派反馈 | 高 | 系统管理员 |
| FR-04 | 任务指派 | 指派反馈给网格员 | 高 | 系统管理员 |
| FR-05 | 网格核查 | 查询指派任务列表 | 高 | 网格员 |
| FR-06 | 网格核查 | 提交实测数据确认 | 高 | 网格员 |
| FR-07 | 数据统计 | AQI趋势统计 | 高 | 决策者 |
| FR-08 | 数据统计 | AQI汇总统计 | 高 | 决策者 |
| FR-09 | 基础数据 | 省市区区域管理 | 中 | 系统管理员 |
| FR-10 | 基础数据 | AQI级别标准维护 | 低 | 系统管理员 |
| FR-11 | 用户管理 | 公众监督员注册 | 中 | 系统管理员 |
| FR-12 | 用户管理 | 网格员账号管理 | 中 | 系统管理员 |

### 4.2 功能需求详述

#### 4.2.1 FR-01 提交AQI反馈

**触发条件**：公众监督员登录后选择"提交反馈"功能

**前置条件**：
- 用户已完成注册且账号状态正常
- 系统中存在网格地址数据

**基本流程**：
1. 用户选择或输入网格地址
2. 用户填写AQI数值（必填）
3. 用户可选填六项污染物指标（PM2.5、PM10、SO₂、NO₂、CO、O₃）
4. 用户点击"提交"
5. 系统校验用户身份
6. 系统保存反馈记录，状态置为"待指派"（status=0）
7. 系统返回反馈ID和当前状态

**异常流程**：
- E1：用户不存在 → 返回"用户不存在"错误
- E2：必填字段缺失 → 返回参数校验错误
- E3：AQI数值超出合理范围（0-500）→ 返回数据范围错误

**业务规则**：
- feedback_time 由系统自动记录为当前时间
- status 初始值为0（未指派）
- assign_grid_member_id 初始值为空

#### 4.2.2 FR-03 查看待指派反馈

**触发条件**：系统管理员进入反馈管理界面

**前置条件**：
- 管理员已登录且角色权限正常

**基本流程**：
1. 系统展示状态为"未指派"（status=0）的反馈列表
2. 列表信息包含反馈ID、网格地址、AQI数值、反馈时间
3. 管理员可按时间、区域筛选

**业务规则**：
- 仅展示未指派的反馈
- 支持分页查询

#### 4.2.3 FR-04 指派反馈给网格员

**触发条件**：管理员在待指派列表中选择某条反馈并点击"指派"

**前置条件**：
- 反馈状态为"未指派"（status=0）
- 目标网格员状态为"启用"

**基本流程**：
1. 管理员选择反馈记录
2. 管理员选择目标网格员
3. 系统校验反馈状态和网格员状态
4. 系统更新反馈记录：assign_grid_member_id 设为目标网格员ID，status 设为1（已指派）
5. 系统返回指派成功信息

**异常流程**：
- E1：反馈不存在 → 返回"反馈记录不存在"
- E2：反馈已被指派 → 返回"该反馈已被指派"
- E3：网格员不存在或已禁用 → 返回"网格员不可用"

**业务规则**：
- 每条反馈仅可指派一次
- 指派后状态不可回退

#### 4.2.4 FR-05 查询指派任务列表

**触发条件**：网格员登录后查看任务列表

**前置条件**：
- 网格员账号状态为启用

**基本流程**：
1. 网格员请求任务列表
2. 系统查询 assign_grid_member_id 等于当前网格员ID的反馈记录
3. 按反馈时间倒序排列
4. 支持分页返回

**业务规则**：
- 仅展示指派给当前网格员的反馈
- 默认每页10条，可配置

#### 4.2.5 FR-06 提交实测数据确认

**触发条件**：网格员对指派任务进行实地核查后提交确认

**前置条件**：
- 反馈已指派给当前网格员
- 反馈状态为"已指派"（status=1）

**基本流程**：
1. 网格员填写实测AQI值（必填）
2. 网格员选填实测PM2.5值
3. 网格员选择确认结果：一致（1）或不一致（2）
4. 系统保存实测数据记录（measure_data表）
5. 系统更新反馈状态为"已确认"（status=2）
6. 同时将实测数据写入统计表（statistics表）

**异常流程**：
- E1：反馈不存在 → 返回"反馈记录不存在"
- E2：反馈未指派给该网格员 → 返回权限错误

**业务规则**：
- measure_time 由系统自动记录
- 每条反馈仅可确认一次
- 确认后反馈状态不可回退

#### 4.2.6 FR-07 AQI趋势统计

**触发条件**：决策者进入统计页面选择趋势查看

**前置条件**：
- 系统中存在已确认的反馈数据

**基本流程**：
1. 决策者选择时间范围（起止日期）
2. 决策者选择区域范围（省级/市级）
3. 决策者选择统计维度（日/月/季度）
4. 系统按条件聚合计算AQI趋势
5. 返回时间序列数据（date、avgAqi、maxAqi、minAqi、count）

**业务规则**：
- 默认统计维度为"日"
- 区域筛选支持省级和市级两级联动
- 月维度格式为YYYY-MM，季度维度格式为YYYY-QN

#### 4.2.7 FR-08 AQI汇总统计

**触发条件**：决策者进入统计页面查看汇总指标

**基本流程**：
1. 决策者选择筛选条件（时间+区域）
2. 系统计算汇总指标：平均AQI、最大AQI、最小AQI、反馈总数
3. 返回汇总数据

**业务规则**：
- 汇总数据与趋势数据使用相同筛选条件
- 平均值保留一位小数

## 七、系统模块划分

### 7.1 模块结构

```
东软环保公众监督平台
├── 公众反馈模块 (public-feedback)
│   ├── 反馈提交
│   └── 反馈状态查询
├── 任务调度模块 (task-dispatch)
│   ├── 待指派反馈查询
│   ├── 反馈指派
│   └── 指派记录查询
├── 网格核查模块 (grid-inspection)
│   ├── 任务列表查询
│   ├── 实测数据提交
│   └── 核查确认
├── 数据统计模块 (data-statistics)
```

## 十、接口需求

### 10.1 接口设计原则

1. 遵循RESTful API设计规范
2. 统一使用JSON格式传输数据
3. 统一响应结构：`{code, message, data}`
4. 使用HTTP状态码语义：200成功、400参数错误、500服务异常
5. 接口路径按角色分组：`/api/public/`、`/api/grid/`、`/api/admin/`、`/api/decision/`

### 10.2 接口清单

## 十二、约束与假设

### 12.1 约束条件

1. 系统基于Spring Boot 2.7 + MyBatis-Plus 3.5技术栈开发
2. 数据库仅支持MySQL 8.0及以上版本
3. 本期不包含用户认证（JWT/Session），后续版本集成
4. 本期不包含前端页面开发，仅提供RESTful API
5. AQI数值范围限定为0-500

### 12.2 假设条件

1. 假设公众监督员已通过其他渠道完成实名注册
2. 假设网格员的网格地址与城市归属关系已预先录入
3. 假设网络环境稳定，客户端与服务端通信正常
4. 假设MySQL数据库服务持续可用

### 12.3 风险识别

| 风险编号 | 风险描述 | 影响程度 | 应对措施 |
|---------|---------|---------|---------|
| R-01 | 高并发反馈提交导致数据库写入瓶颈 | 中 | 引入消息队列异步写入 |
| R-02 | 统计查询数据量大导致响应缓慢 | 高 | 建立索引 + 定时预聚合 |
| R-03 | 网格员未及时处理指派任务 | 中 | 增加超时提醒机制 |
| R-04 | 公众恶意提交大量虚假反馈 | 中 | 增加频率限制 + 信用评分 |

## 附录A：AQI级别标准参考

| AQI范围 | 级别 | 颜色 | 健康影响 |
|---------|------|------|---------|
| 0-50 | 优 | 绿色 | 空气质量令人满意，基本无空气污染 |
| 51-100 | 良 | 黄色 | 可接受，某些污染物可能对极少数敏感人群有较弱影响 |
| 101-150 | 轻度污染 | 橙色 | 易感人群症状轻度加剧，健康人群出现刺激症状 |
| 151-200 | 中度污染 | 红色 | 进一步加剧易感人群症状，可能影响健康人群心脏和呼吸系统 |
| 201-300 | 重度污染 | 紫色 | 心脏病和肺病患者症状显著加剧 |
| 301-500 | 严重污染 | 褐红色 | 健康人群有明显强烈症状 |
